# Code Style Guidelines

- 처음부터 완벽한 아키텍처보다 지금 필요한 변화만, 읽히게 만든다.
- 과한 래핑/추상화/방어코드로 인해 파악이 어려워지는 것을 방지한다.
- 광범위한 포맷팅/네이밍 변경/파일 이동/대규모 함수 분리는 금지.
- 새 함수는 중복이 2회 이상 생길 때만 만든다.
- 나중에 바뀔 수도라는 이유로 인터페이스/플러그인 구조 만들지 말 것.
- 새 방어코드는 I/O 경계(파일/네트워크/외부 API/사용자 입력) 에만 추가.
- 내부 로직에 try/except 남발 금지.
- 기존 에러 처리 수준이 있다면 그 수준에 맞춘다.
- 오타/죽은 코드 1~2줄/명백한 버그 수정
- 건드린 코드 근처의 아주 작은 개선만 허용:
- 호출 깊이 2단계 이하 (A→B→C 정도까지만)
