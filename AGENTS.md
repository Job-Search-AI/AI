# Code Style Guidelines

- 처음부터 완벽한 아키텍처보다 지금 필요한 변화만, 읽히게 만든다.
- 과한 래핑/추상화/방어코드로 인해 파악이 어려워지는 것을 방지한다.
- 광범위한 포맷팅/네이밍 변경/대규모 함수 분리는 금지.
- 네이밍은 너무 길고 복잡하지 않게 직관적이고 간단하게 짓는다.
- 새 함수는 중복이 2회 이상 생길 때만 만든다.
- 나중에 바뀔 수도라는 이유로 인터페이스/플러그인 구조 만들지 말 것.
- 새 방어코드는 I/O 경계(파일/네트워크/외부 API/사용자 입력) 에만 추가.
- 내부 로직에 try/except 남발 금지.
- 기존 에러 처리 수준이 있다면 그 수준에 맞춘다.
- 오타/죽은 코드 1~2줄/명백한 버그 수정
- 건드린 코드 근처의 아주 작은 개선만 허용:
- 호출 깊이 2단계 이하 (A→B→C 정도까지만)
- 모든 작업이 마무리되면, 커밋메시지를 아래와 같은 규칙으로 한글로 작성해준다.
  - feat: 새로운 기능 추가
  - add : feat 이외의 부수적인 코드, 라이브러리 등을 추가한 경우, 새로운 파일(Component나 Activity 등)을 생성한 경우도 포함
  - fix: 버그 수정
  - docs: 문서 수정
  - style: 코드 포맷팅, 세미콜론 누락, 코드 변경이 없는 경우
  - refactor: 코드 리팩토링
  - test: 테스트 코드, 리팩토링 테스트 코드 추가
  - chore: 빌드 업무 수정, 패키지 매니저 수정, production code와 무관한 부분들 (.gitignore, build.gradle 같은)
  - comment: 주석 추가 및 변경
  - remove: 파일, 폴더 삭제
  - rename: 파일, 폴더명 수정
