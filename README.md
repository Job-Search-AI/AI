## 프로젝트 계기 및 소개

기존 채용 사이트들은 직무, 지역, 경력 등 단순 키워드 기반 검색만 제공을 합니다.

취업준비생이 원하는 근무 조건이나 환경을 정확히 찾기위해서는 채용공고를 직접 들어가 확인해야 했습니다.

예를 들어 '재택 가능', '자율 출퇴근제'와 같은 근무 조건은 사용자가 **일일이 공고를 열어 확인해야만 알 수 있어 비효율적**이였습니다.

<br>

**이런 불편함을 해결하기 위해, 사용자가 자연어로 자신의 희망 조건을 입력하면 AI가 의미를 이해하고,** 

**사용자가 희망 조건을 입력하면 채용공고를 검색후 분석해 조건에 맞는 공고를 추천하도록 LLM 기반 챗봇 시스템을 설계 했습니다.**

<br>

이를 통해 원하는 근무 환경까지 쉽게 확인할 수 있고, **검색 과정에서의 불필요한 반복을 줄이며 보다 빠르게 자신에게 맞는 회사 정보**를 얻을 수 있게 됩니다.

<br>

## 개발 환경

- 크롤링 : Selenium, BeautifulSoup
- AI : Pytorch, BERT, CRF, FAISS, Qwen2.5(skt/A.X-4.0-Light), Snowflake embedding model, Hybrid Retrieve(BM25 + Embedding model),
- Front : React, Tailwind Css
- 버전 관리 : Github, Conda
- 배포 환경 : Fast API, GCP VM

<br>

## **채택한 개발 기술**

### KoBERT + CRF

| **Model** | **Accuracy** |
| --- | --- |
| [BERT base multilingual cased](https://github.com/google-research/bert/blob/master/multilingual.md) | 0.875 |
| KoBERT | [**0.901**](https://github.com/SKTBrain/KoBERT/blob/master/logs/bert_naver_small_512_news_simple_20190624.txt) |
| [KoGPT2](https://github.com/SKT-AI/KoGPT2) | 0.899 |

BERT 모델을 사용하여 한국어 개체인식은 skt/KoBERT가 성능이 좋아 선택했습니다.

처음에는 skt/KoBERT만 사용 하였지만, CRF를 같이 사용하여 문맥을 이해한뒤 개체인식을 하여 더 좋은 성능을 만들었습니다.

<br>

### Hybrid Retrieve(BM25 + Embedding Model)

처음은 BM25을 사용했지만, 사용자의 의미적 문맥을 이해하지 못해 올바른 검색을 하지 못했습니다.

Embedding model, 쿼리 전처리, BM25 키워드 검색, 하이브리드 검색 기법을 도입하여 검색 정확도를 크게 향상시켰습니다.

| 평가 지표 | 기존 시스템 | 하이브리드 시스템 | 개선율 | 목표 달성 |
| --- | --- | --- | --- | --- |
| **Recall@1** | 0.164 | 0.187 | **+14.0%** | ✓ |
| **Recall@3** | 0.349 | 0.531 | **+52.1%** | ✓ |
| **Recall@5** | 0.430 | 0.625 | **+45.3%** | ✓ |
| **Recall@10** | 0.683 | 0.643 | -5.9% | △ |
| **Precision@1** | 0.455 | 0.545 | **+19.8%** | ✓ |
| **Hit@1** | 0.455 | 0.545 | **+19.8%** | ✓ |
| **Hit@3** | 0.818 | 0.909 | **+11.1%** | ✓ |
| **Hit@5** | 0.818 | 1.000 | **+22.2%** | ✓ |

**중간 순위 성능 45-52% 향상**: 실용적 검색 범위에서 큰 성능 향상

**시스템 안정성**: 평균 응답 시간 3초 이내 유지

<br>

이번 개선 작업을 통해 채용 정보 검색 시스템의 성능이 크게 향상되었으며, 사용자에게 더 정확하고 관련성 높은 검색 결과를 제공할 수 있게 되었습니다. 

향후 지속적인 개선을 통해 더욱 완성도 높은 검색 시스템으로 발전시켜 나갈 예정입니다.

<br>

### skt/A.X-4.0-Light

개인 프로젝트로 **모델의 크기가 작으면서 한국어 성능이 좋은 모델**을 찾는것이 목표였습니다.

![image.png](attachment:56240c2c-3fec-495b-8ec1-fed26de56af3:image.png)

[LLM 한국어 리더보드](https://leaderboard.dnotitia.com/) 에서 [naver-hyperclovax/HyperCLOVAX-SEED-Think-14B](https://huggingface.co/naver-hyperclovax/HyperCLOVAX-SEED-Think-14B) 은 14.7B로 크기가 작으며 한국어 성능이 가장 좋고 특히 **한국어 성능 최적화**에 집중한 모델로서 학습 효율성이 좋을것이라 판단했습니다.

하지만 GPU T4 기준으로 Vram 16GB 메모리를 넘어서는 사용량으로 **제한된 환경으로 다른 모델**을 찾게 되었습니다.

<br>

[skt/A.X-4.0-Light](https://huggingface.co/skt/A.X-4.0-Light)는 SK텔레콤이 출시한 한국어 특화 대형언어모델(LLM)입니다.

**7B 크기로 작아, 16GB 메모리에도 사용 가능하며, LLM 리더보드에서도 sLLM 중에서도 좋은 성능**을 보여주었습니다.

또한 Qwen2.5 오픈소스 모델을 기반으로 하여 대규모 한국어 데이터셋으로 추가 학습을 진행해 한국어 이해에 좋은 모델이여서 선택했습니다.

<br>

HyperCLOVAX-SEED-Think-14B을 LLM 모델로 사용했으나, GPU T4 기준으로 Vram 16GB 메모리를 넘어서 제한된 환경으로 skt/A.X-4.0-Light 모델을 사용해 최대 12GB 메모리 사용으로 최적화하였습니다.

<br>

## 성장 경험

**사용자 질문 기반 LLM을 통해 URL 변환**

사용자의 질문을 채용공고 URL로 변환하여 실시간으로 크롤링하는 과정에서는 처음에 LLM을 활용한 프롬프트 템플릿 방식을 시도했습니다.

예상보다 정확하게 URL을 생성하지 못했고, 이를 개선하기 위해 프롬프트 엔지니어링을 공부하고 적용하면서 성능을 점진적으로 향상시킬 수 있었습니다. 

하지만 LLM만으로는 URL 변환에는 성능적으로 한계가 존재했습니다.

<br>

**BERT 기반 NER과 커스텀 모델 구현**

BERT 기반 NER 개체 인식을 활용하여 URL을 생성하는 방법을 적용했습니다. 

SKT/kobert를 사용하였으며, Hugging Face에서 제공되는 자료가 충분하지 않아 CRF와 필요한 개체만 인식하도록 모델을 만들기 어려웠습니다.

그래서 다른 프로젝트의 개체 인식 코드를 분석하고 공부했습니다.

결국 개체인식의 전체 워크플로우를 이해하게 되었고 제 프로젝트에 맞게끔 적용했습니다.

또한 CRF를 추가하여 문맥에 맞는 성능을 향상시켰습니다.

<br>

**하이브리드 리트리버 학습과 성능 향상**

마지막으로, 임베딩과 리트리버를 결합한 하이브리드 리트리버 개념을 새롭게 학습하고, 이를 BM25와 임베딩 검색에 적용했을때, 가장 크게 성능이 향상된것을 체감했습니다.

이번 프로젝트를 통해 데이터 수집, 전처리, 개체 인식, 정보 검색까지 전ㅍ반적인 워크플로우를 이해하고 실무에 적용할 수 있는 역량을 갖추게 되었다고 생각합니다.
